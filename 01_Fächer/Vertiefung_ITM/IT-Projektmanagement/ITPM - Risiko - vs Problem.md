# ITPM - Risiko - vs Problem

> [!info] Kurzdefinition
> Während das Eintreten eines Risikos immer **unsicher** ist, ist ein Problem bereits eingetreten oder wird sicher eintreten. Die Unterscheidung ist für das Risikomanagement wesentlich. Maßnahmen unterscheiden sich grundlegend: Probleme erfordern korrektive, Risiken präventive Maßnahmen.

## Beschreibung

Aus Folien 36–37:

Für das Risikomanagement ist die Unterscheidung zwischen Problemen und Risiken **wesentlich**:

- **Risiko:** Unsicher, kann vielleicht eintreten.
- **Problem:** Tatsache, Sicherheit (ist vorhanden oder eingetreten).

## Bestandteile / Aufbau

### Prüfschema (Folie 36)

```
Ereignis →
  Unsicher? → 
    Ja → Verlust möglich? →
      Ja → Dem Projekt zuordenbar? →
        Ja → Risiko
        Nein → Geschäftsrisiko (außerhalb Projekt)
      Nein → Kein Einfluss
    Nein → Problem
```

Die zweite und dritte Abfrage dienen dazu, festzustellen, ob ein Risiko **im Projekt behandelt** werden muss:

- **Kein Schaden zu erwarten** → Risiko muss nicht behandelt werden.
- **Risiko unabhängig vom Projekt** → als **Geschäftsrisiko außerhalb des Projekts** behandeln.

### Vergleich Problem vs. Risiko (Folie 37)

| Aspekt | Problem | Risiko |
|---|---|---|
| **Auswirkung** | Ist eingetreten | Kann eintreten |
| **Maßnahme** | Korrektiv | Präventiv |
| **Denkweise** | Taktisch | Strategisch |
| **Zeithorizont** | Kurz | Lang |
| **Zeit** | Vergangenheit | Zukunft |

## Abgrenzung

- **Problem** ≠ **Risiko**: zentrales Unterscheidungskriterium ist die **Sicherheit/Unsicherheit** des Eintretens.
- **Geschäftsrisiko** ≠ **Projektrisiko**: Geschäftsrisiken werden außerhalb des Projekts behandelt.
- Beziehung zu **Issue Management**: Probleme werden als „Issues" verfolgt, Risiken als „Risks".

## Prüfungsrelevanz

**Typische Definitionsfrage:** Worin unterscheiden sich Risiko und Problem entlang der fünf Kriterien?

**Anwendungsfrage:** Ein Server fällt aus → ist das ein Risiko oder ein Problem? Welche Maßnahme passt?

**Diskussionsfrage:** Wann wird ein Risiko zum Problem? Welche Übergangsmechanismen kennt man?

**Mögliche Anschlussfragen:**
- Was tun, wenn ein Risiko nicht dem Projekt zuordenbar ist?
- Wie verhalten sich Issues zu Problemen?
- Welche Werkzeuge unterstützen Issue-/Risk-Tracking?
