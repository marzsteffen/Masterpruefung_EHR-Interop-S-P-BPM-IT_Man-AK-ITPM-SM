# ITPM - Agile - User Story

> [!info] Kurzdefinition
> Eine User Story beschreibt eine Anforderung kurz aus Kundensicht. In Scrum sind User Stories ein typisches Format für Product-Backlog-Einträge. Standardformat laut Vorlesung: „Als [Kundentyp] [möchte] ich, [damit]."

## Beschreibung

Die Vorlesung erwähnt **Use Cases** als „Anwendungsfälle, die Anforderungen aus Kundensicht beschreiben" (Folie 15) und führt **User Stories** im Kontext des Product Backlog ein (Folien 39–41). Sie werden auf Story Cards festgehalten.

**Standardformat (Folie 41):**

> „Als [Kundentyp] [möchte] ich, [damit]."

## Bestandteile / Aufbau

Beispiele aus der Vorlesung (Folie 40):

**Gute User Stories** (kundenorientiert, funktional):
- „Als Benutzer kann ich nach Stellen suchen."
- „Als HR-Mitarbeiter kann ich neue Stellenangebote sehen."

**Schlechte User Stories** (technisch / kosmetisch, ohne Kundennutzen):
- „Die Software soll in Java geschrieben werden." *(Technologievorgabe, kein Nutzen)*
- „Der Button ist grün und blinkt immer wieder." *(reine Implementierungs-Detailvorschrift)*

## Beispiel

Übungsaufgabe der Vorlesung (Folie 41): Für „Ein gemeinsames Büro/Arbeitszimmer zu Hause einrichten" mindestens 8–10 User Stories formulieren.

## Abgrenzung

- **Epic** (siehe [[ITPM - Agile - Techniken Übersicht]]): Zusammenfassen verwandter Anwendungsfälle in größerer Granularität.
- **Use Case**: detaillierterer Anwendungsfall mit Akteuren, Vorbedingungen, Ablauf etc. – im PDF nur als Begriff erwähnt, ohne Detailausführung.
- **Funktionale Spezifikation** (klassisch): umfassend und vorab; User Stories sind kurz und entstehen iterativ.

## Prüfungsrelevanz

**Typische Definitionsfrage:** Welches Standardformat hat eine User Story laut Vorlesung?

**Anwendungsfrage:** Formulieren Sie für ein Patientenportal drei User Stories aus Sicht eines Patienten und drei aus Sicht eines Arztes.

**Diskussionsfrage:** Warum sind technische Vorgaben („Java", „grüner Button") schlechte User Stories? Welche Strukturierung passt für Nicht-Funktional-Anforderungen?

**Mögliche Anschlussfragen:**
- Wie unterscheiden sich User Story und Use Case?
- Welche INVEST-Kriterien (Independent, Negotiable, Valuable, Estimable, Small, Testable) gibt es für gute User Stories? *(im PDF nicht definiert)*
- Wer schreibt die User Stories – Product Owner, Team, Kunde?
