# ITPM - Extreme Programming

> [!info] Kurzdefinition
> Extreme Programming (XP) ist eine agile Methode speziell für die Programmierung. Sie nähert sich den Anforderungen des Kunden in kleinen Schritten mit täglichen Abstimmungen und engem Kunden-Einbezug. Charakteristisch: Pair Programming.

## Beschreibung

Laut Vorlesung (Folie 13) wird Extreme Programming **speziell in der Programmierung verwendet**. Es ist eine **Annäherung an die Anforderungen des Kunden in kleinen Schritten** mit **täglichen Abstimmungen** und **Einbezug des Kunden**.

Teilweise kommt **Pair Programming** zum Einsatz.

## Bestandteile / Aufbau

Im PDF werden nur die Kernmerkmale genannt:

- Inkrementelle Annäherung an Anforderungen
- Tägliche Abstimmungen
- Kunden-Einbezug
- Pair Programming (teilweise)

*Detailliertere XP-Praktiken (Test-Driven Development, Continuous Integration, Refactoring, Kollektives Code-Eigentum, einfaches Design, Coding Standards, etc.) werden im PDF nicht aufgeführt.*

## Abgrenzung

- **Scrum** fokussiert auf Projektmanagement-Framework, XP auf **Programmierpraktiken**. Beide werden oft kombiniert (Scrum für Prozess, XP für Code-Qualität).
- **FDD** und **Crystal** sind ebenfalls agile Methoden, aber mit anderen Schwerpunkten (siehe [[ITPM - Feature Driven Development]], [[ITPM - Crystal Family]]).

## Prüfungsrelevanz

**Typische Definitionsfrage:** Was ist das charakteristische Merkmal von Extreme Programming? Welche Praxis sticht besonders hervor?

**Anwendungsfrage:** Wann ist XP sinnvoll – Scrum, XP, oder beides kombiniert?

**Diskussionsfrage:** Pair Programming verdoppelt scheinbar die Personenstunden – warum kann es trotzdem effizient sein?

**Mögliche Anschlussfragen:**
- Welche XP-Praktiken sind im PDF nicht erwähnt, aber im Standardrepertoire? *(im PDF nicht definiert: TDD, CI, Refactoring, etc.)*
- Welche Rolle spielt der Kunde in XP – ist er „on-site"?
